## FinAgent Pro 全面代码审计报告与冠军优化方案

审计日期: 2026-06-10 | 项目: AFAC2026 方向四 Agentic AI 参赛作品

---

### 一、项目整体评价

FinAgent Pro 是一个架构概念优秀的多Agent智能投顾系统——编排器(Orchestrator) + 四位AI专家链式协作、WebSocket实时推送Agent思考过程、国产大模型全链路自主可控。这些设计理念在参赛作品中具有很强的差异化竞争力。

然而，深入审计后发现项目在 **安全性、正确性、可靠性** 三个维度存在系统性缺陷。后端共发现 7 个严重问题、8 个高危问题；前端 3 个严重问题、6 个高危问题；运维部署 4 个严重问题、5 个高危问题。如果不加以修复，这些问题在演示现场有较高概率导致功能失效或数据泄露。

以下按优先级分为 **P0 (必须修复)**、**P1 (强烈建议)**、**P2 (锦上添花)** 三个层级。

---

### 二、P0 级问题 — 不修复则演示必崩或有安全风险

**【后端-C1】技术指标接口必定崩溃**
`services/market_data.py` 的 `calculate_technical_indicators` 函数返回 Dict 摘要值，但 `main.py` 的股票历史数据端点把它当作 DataFrame 使用，导致 `GET /api/market/stock/{symbol}` 每次调用都抛出 KeyError 或 AttributeError。这是演示中最核心的K线图数据接口，必须修复返回类型，让函数同时返回修改后的 DataFrame 和摘要。

**【后端-C2】LLM/AKShare 同步调用阻塞整个事件循环**
`base_agent.py` 中 `run_llm` 调用的是同步 `self.llm.invoke()`，虽然标记为 async 但实际是阻塞调用。每次 LLM 推理（2-10秒）期间，整个 asyncio 事件循环被冻结，WebSocket 消息无法推送、健康检查超时、其他请求无法处理。AKShare 的所有网络调用同样如此。修复方案：`run_llm` 改用 `await self.llm.ainvoke()`；AKShare 调用包装为 `await asyncio.to_thread()`。

**【后端-C3】Orchestrator 回调累积导致跨用户数据污染**
Orchestrator 是模块级单例，每次请求都追加一个 progress callback 到 `_progress_callbacks` 列表，但从未清理。N 次请求后：所有 N 个回调同时触发，用户 A 的 WebSocket 会收到用户 B 的分析进度，内存持续增长。修复方案：每次请求完成后在 finally 块中移除对应 callback，或改用 per-request callback 而非全局列表。

**【后端-C4】路径穿越漏洞**
`SessionMemory` 的 `session_id` 直接拼接到文件路径中，无任何校验。攻击者可通过 `..%2F..%2Fetc%2Fcrontab` 等值读写任意文件。修复方案：对 session_id 进行正则校验（仅允许 `[a-zA-Z0-9_-]`）或使用 `werkzeug.utils.secure_filename`。

**【后端-C5】Docker 镜像中嵌入敏感信息**
`Dockerfile.backend` 无条件 COPY .env 到镜像中，且项目缺少 `.dockerignore` 文件。所有 API 密钥（DeepSeek、智谱、JWT Secret）被永久嵌入镜像层。修复方案：添加 `.dockerignore` 排除 `.env`、`.git`、`node_modules` 等；Dockerfile 中移除 COPY .env 行；运行时通过 docker-compose 的 `env_file` 注入。

**【后端-C6】数据库/Redis 端口暴露 + 容器以 root 运行**
docker-compose 将 PostgreSQL(5432) 和 Redis(6379) 端口绑定到宿主机 0.0.0.0，Redis 无密码认证，后端容器以 root 身份运行。修复方案：移除 ports 映射（仅保留内部网络通信）；为 Redis 设置 `requirepass`；Dockerfile 添加 `USER` 指令切换为非 root 用户。

**【前端-C7】WebSocket 无限重连/重渲染**
`useWebSocket` hook 的 `connect` 回调依赖的 options 对象在 App.tsx 中每次渲染都创建新引用，导致 useEffect 在每次状态变更时断开并重连 WebSocket。在分析过程中（Agent 推送消息最频繁时），WebSocket 连接不断被拆除重建，消息大量丢失。修复方案：用 `useMemo` 稳定 options 对象，或在 hook 内部用 `useRef` 保存最新的 callbacks。

**【前端-C8】HTTP 响应与 WebSocket 消息竞态条件**
`runAnalysis` 函数在发送 HTTP POST 后立即在 finally 块中清空 sessionId，触发 WebSocket 断开。HTTP 响应和 WebSocket 的 `final_report` 消息都会设置 `analysisResult`，产生 last-write-wins 竞态。修复方案：等待 WebSocket 的 `final_report` 消息作为唯一结果源，HTTP 响应仅作为触发确认；或在 WebSocket 确认收到所有消息后再断开。

### 三、P1 级问题 — 影响演示质量和评委印象

**【后端-H1】最终投资建议逻辑过于简单**
Orchestrator 的 `synthesize_report` 中，投资建议仅用 `risk_level < 60 ? "buy" : "hold"` 一行代码生成，置信度硬编码为 0.78。四个 Agent 的深度 LLM 分析在最终输出中完全被忽略。修复方案：让组合顾问的 LLM 输出直接驱动建议文本和置信度，或基于多 Agent 输出生成综合评分。

**【后端-H2】组合风险计算数学错误**
`market_tools.py` 的 `get_portfolio_risk` 将各股票加权收益率拼接成一维数组后求标准差，完全忽略了协方差矩阵。正确的组合方差公式是 `σ² = ΣΣ wᵢwⱼCov(rᵢ,rⱼ)`，当前方法无法体现分散化效应。修复方案：使用 numpy 构建收益率矩阵，计算协方差矩阵后按公式求组合波动率和 VaR。

**【后端-H3】异常处理基础设施完全未启用**
`exception_handlers.py` 定义了 `FinAgentException` 等异常类和 `setup_exception_handlers()` 函数，但 main.py 从未调用该函数。所有错误处理都用 `except Exception` + `HTTPException(500)`。修复方案：在 main.py 中调用 `setup_exception_handlers(app)`，各模块抛出对应的自定义异常。

**【后端-H4】Agent 异常时向前端泄露完整堆栈**
四个 Agent 的异常处理都用 `traceback.format_exc()` 生成内容，完整的 Python 堆栈（含内部路径、行号、变量值）通过 WebSocket 推送给前端。修复方案：替换为 `logger.exception()` 记录日志，返回用户友好的错误消息。

**【后端-H5】请求参数缺乏校验**
`OrchestratorRequest` 的 `symbols` 接受空列表（导致 ZeroDivisionError）、`investment_amount` 接受负数、`risk_preference` 接受任意字符串。修复方案：Pydantic Field 添加 `min_length=1`、`gt=0`、`Literal["conservative","moderate","aggressive"]`。

**【后端-H6】requirements.txt 包含 smtplib（标准库）**
`pip install -r requirements.txt` 会因此报错失败。同时 celery、nest-asyncio、prometheus-client、opentelemetry 系列、finnhub-python、slowapi、jinja2 共 8 个依赖从未被代码引用。修复方案：移除 smtplib 行；清理或注释掉未使用依赖。

**【后端-H7】Dockerfile.backend 缺失 15+ 关键依赖**
Dockerfile 中 pip install 的包列表与 requirements.txt 严重不一致，缺少 asyncpg、sqlalchemy、alembic、redis、celery、python-jose、bcrypt 等核心依赖。构建出的镜像无法启动。修复方案：Dockerfile 改为 `COPY requirements.txt && pip install -r requirements.txt`。

**【前端-H8】仪表盘行情数据为前端随机数**
App.tsx 的 `fetchMarketOverview` 用 `Math.random()` 生成恒生指数、科技指数和成交额。API 响应仅作为"成功"标志使用，实际展示的数值全是伪造的。演示时评委若注意到数值每次刷新都随机跳动，将严重损害可信度。修复方案：后端提供真实市场概览 API，前端使用返回数据。

**【前端-H9】数字员工工作台工具日志永远为空**
`toolCalls` 状态初始化为空数组后再无写入，OrchestratorWorkbench 的工具调用日志面板始终显示空表格。这是演示脚本中重点提到的功能。修复方案：在 WebSocket 消息处理中解析 `tool_call` 类型消息并更新 `toolCalls` 状态；后端 Agent 调用工具时发送对应消息。

**【前端-H10】App.tsx 上帝组件 (493行)**
整个应用——状态管理、WebSocket、HTTP 请求、导航、7 个页面渲染——全在一个组件中。子视图定义为闭包函数，无法独立测试，任何状态变更都触发全量重渲染。修复方案：提取 Dashboard、AgentChat、Workbench、StockList 等为独立组件。

**【前端-H11】零测试覆盖**
前后端均无任何 .test 或 .spec 文件。对于声称"生产就绪"的参赛作品，这是一个明显短板。修复方案：至少为后端核心流程（orchestrator.run、各 agent.analyze）和前端关键组件编写基础测试。

**【前端-H12】react-app-env.d.ts 覆盖 Antd 真实类型为 any**
该文件将 antd 和 @ant-design/icons 的所有组件重声明为 `React.FC<any>`，完全遮蔽了 antd@5.20 自带的类型定义。所有 UI 组件的 props 失去类型检查。修复方案：删除该文件中的 antd 模块声明。

**【前端-H13】缺少 React ErrorBoundary**
无任何错误边界组件。如果 ECharts 渲染异常数据，整个应用白屏崩溃。修复方案：在 index.tsx 中包裹顶层 ErrorBoundary。

### 四、P2 级问题 — 提升品质感和专业度

**【M1】市场数据逻辑重复**
`market_tools.py` 和 `market_data.py` 独立实现了相同的技术指标计算（MA、MACD、RSI、布林带），bug 修复需改两处。建议统一到 `market_data.py`，`market_tools.py` 作为薄代理调用。

**【M2】情绪分析本质上是 RSI 查表**
SentimentScanner 仅根据 RSI 数值映射情绪标签，无真实新闻/社媒情绪分析。finnhub-python 虽在依赖中但从未使用。建议至少添加基于 AKShare 资金流向数据的市场宽度指标。

**【M3】硬编码股票名称映射仅 10 只**
`market_data.py` 中仅映射 10 只港股名称，akshare 的 spot 数据本身包含名称字段。建议直接使用 akshare 返回的名称。

**【M4】ECharts 图表每次数据变更都 dispose + init**
三个图表组件在 useEffect 中对 ECharts 实例调用 dispose 再 init，而非用 setOption 更新。建议改为 init 一次 + setOption 更新。

**【M5】WebSocket 重连无退避策略**
前端 WS 断线后固定 3 秒重连，无指数退避、无抖动、无最大重试次数。建议实现 1s→2s→4s→8s 指数退避，上限 5 次。

**【M6】Nginx 缺少安全头和超时配置**
无 X-Content-Type-Options、X-Frame-Options、CSP 等安全头；WebSocket proxy 缺少 proxy_read_timeout（默认 60 秒后断开）；无 gzip 压缩。

**【M7】部署指南多处信息不准确**
Python 版本写 3.9+ 但实际用 3.10；docker build 命令的 context 路径错误；未提及数据库迁移步骤；云服务器推荐配置(2核4GB)低于 compose 实际资源需求(4.25GB+)。

**【M8】print() 替代正式日志**
多处使用 print() 输出调试信息，但 loguru 已是项目依赖。建议全局替换为 logger.info/debug。

**【M9】CSS 响应式过度覆盖**
`@media (max-width: 1200px)` 用 `!important` 强制所有 ant-col 100% 宽度，破坏所有多列布局，13 寸笔记本也会触发。

**【M10】前端无障碍缺失**
无 ARIA 标签、无键盘导航、颜色状态指示器缺少文字替代、label 未关联 input。

### 六、冠军品质加分项

如果时间充裕，以下改进可以显著提升评委印象：

**演示体验优化：** 在 Agent 分析过程中增加"思考中..."的动态动画和进度百分比，而非简单的文字滚动。参考 ChatGPT 的逐步输出效果，让每个 Agent 的分析过程有视觉节奏感。

**可解释性增强：** 在最终报告中增加"决策溯源"板块——用可视化连线展示每个 Agent 的输出如何影响最终建议，比如"市场分析师判定偏多 → 风险经理评估VaR 2.3% → 综合建议: 买入"。这正是评委关注的 Agentic AI 可解释性亮点。

**实时数据可信度：** 在行情面板旁增加"数据来源: AKShare 实时行情"的小标签和时间戳，让评委明确感知到数据的真实性。

**一键演示模式：** 增加一个"演示模式"按钮，自动执行预设的分析流程（腾讯00700、10万港币、稳健型），确保演示时操作零失误。

**性能指标展示：** 在数字员工工作台中增加"本次分析耗时 X 秒、调用 Y 次工具、处理 Z 条数据"等指标，展示系统的工程成熟度。
